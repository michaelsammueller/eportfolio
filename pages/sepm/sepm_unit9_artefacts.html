e-Portfolio Activity 1: Scope Creep

Task: Answer the following questions:

What is scope creep?

The term scope creep refers to the phenomenon when the scope increases during the project, and eventually includes things which were never meant to be part of the project. As the scope increases, the price of the project will increase and so will the time needed to complete the project.

What aspects of a project are dependent on scope creep?

Time and price, which will affect the number of resources needed and might affect the quality of the project.

How to balance the competing requirements when scope creep occurs?

Communication is key. If the scope increases and more time is needed, then an analysis needs to be done to determine whether the project allows for such a delay to occur.

How do you stop scope creep?

Again, clear communications. As a project manager, you should clearly communicate as to what the scope is and where expectations should be, in order to avoid scope creep from occurring.

e-Portfolio Activity 2: Improving Code Quality

Task: Refer to the Mertz (2019) resource. Use some Python code which you have developed in the past and apply at least 3 of the strategies presented at the source to improve its quality.

After reading through the resource, I dediced to pick 4 strategies which stood out to me:

  1. Indentation: Use 4 spaces per indentation level instead of tabs.
  2. Line Length: Keep a consistent line length and do not exceed a character limit of 99 characters for code, and 72 characters for comments.
  3. Arguments: Try to limit your functions to 2 or 3 arguments. Do not exceed line length - if you call a function that requires multiple arguments spanning multiple lines, add an additional layer of indentation.
  4. Imports: Imports should be at the top of the file in the following order: Standard Packages, Third-Party Packages, and Local Packages. Refrain from using "import *" statements, and do not chain package imports.

I picked these four strategies because I think they are among the most relevant. Readability of code is of utmost importance, especially when working in a team. If a software engineer, who had nothing to do with the project, were to look at the code, they should be able to at least understand what is going on, even if they don’t necessarily know why. The “why” question should be answered by comments, which is why documenting code thoroughly is also very important.

I looked through some of my code files and found that most of them adhered to these strategies mentioned above, thanks to the IDE I use, but also because I have made it a habit to use pylint and flake8 to test my code after I have written it.
