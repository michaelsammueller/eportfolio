Unit 5: Reflection & Artefacts

Plan

I've been using these past few days to work on my submission for the next assignment. I enjoy coding and solving problems, so this one is right up my alley. First, I decided to quickly sketch out what I think my application should include. After reading over the requirements, I decided on the following:

  • GUI: I want my user to be able to interact with the application via a graphical user interface. Especially when it comes to programs that visualise something, I like to work with GUIs. The best way to do this, for an application this size, within Python, will be to use Tkinter, a library which I have worked with before.
  • TreeView: I want my application to have a tree view, which will allow the user to see the hierarchy of the tree and assign costs/values to each node. The program should have a "calculate" button, which, when clicked, should calculate the minimum and maximum cost associated with this particular tree. Thsi way, the user can interact with the tree before generating a visual representation of it.
  • Visualisation: The program should include a "visualise" button, which, when pressed, should generate a visual representation of the attack tree, based on the attack tree specification. I also want this visualisation to include the minimum and maximum values calculated above, as part of a legend of some sort. To achieve this, I am planning to use the matplotlib and networkx libraries.
  • JSON: I have decided to work with JSON files, as I have worked with them before as part of my assignment in the 'Secure Software Design' module. I have created an outline for an attack tree specification, which I want my attack trees to use. I have tried to find an industry standard format online, but besides file formats, it seems like this varies from industry to industry. I have therefore attempted to keep my JSON format relatively straight forward, so that it can be easily adjusted at a later point in time.
  • Object-Oriented: Although it is a relatively small application, in terms of lines of code, I like to follow an object-oriented paradigm. This means I will try separate parts of the code into different classes, improving the maintainability and reusability of the code. This also makes it easier for me to work on the code, as I will immediately know where to find specific parts of the code whenever a bug occurs during debugging.

Summary: If everything works out the way I have planned it above, the user will be able to load a JSON file into the application, and will be presented with an hierarchical tree view of the attack tree. Within this tree view, the user will be able to assign values or costs to each node, and calculate a mininum and maximum cost/damage associated with this attack tree. The user can then ask the program to visualize the tree. The program will generate an attack tree graph based on the JSON file, with a legend that includes those 'minimum' and 'maximum' values calculated in the previous step. The user can then safe this graph as a '.png' file on their machine, to share with other stakeholders.

Structure

To complete the steps described above, I propose the following structure for my code:

  1. First, I will create a 'master' file (main.py), which I will use to create a prototype of the application. Within this file, I will create a working version of the application, allowing me to experiment with the code. This also means that, if I were to run out of time later, I would have a working version of the application.
  2. Decision point - There are two ways I can approach the next step. I can either separate the classes ('GUI', 'File', 'Graph') within the same file, or create a file for each one ('gui.py', 'file.py', 'graph.py'). Since the code is relatively small for this type of application, both structures would work. I will see which one is easier to implement.
  3. For the last step, I want to package my application so it can be run via a '.exe' or '.dmg' file directly on the user's machine. This vastly improves the user experience, especially for non-technical users.
