UML models within the Software Development Life Cycle

Task: Discuss which UML models are most applicable at different stages of the Software Development Life Cycle.

Based on what I have learned in Unit 3 so far, UML consists of the following three model types:

Behavioural: Capture the interactions between system components over time.

Structural: Represent components of a system and relationships between them.

Architectural Capture a high-level perspective of the system.

To initially create a basic sketch of a system in order to outline ideas, architectural models are helpful as they provide a high-level perspective of the system. This is especially useful if it is done together with a customer who might not be as technically literate; however, the benefits don’t end there. It also allows a product designer/developer to have an initial, basic understanding of the system, which will allow them to start planning the system in more detail. This, however, is not the only type of model that can be helpful/useful at this stage. Behavioural models such as “Use Case” diagrams allow for very basic initial sketching of how a system might be used by one of its actors. For example, if one is developing a leave management program, looking at it from a user’s point of view might prove useful for understanding how it works and for classifying objects, behaviours and methods.

After the initial overview, structural models appear to be the logical consequence. This allows a team to design and understand the components of a system and the relationships between them.

Thirdly, behavioural models allow the team to design the interactions between systems and components and understand how their behaviour will change over time.

Additional Activities

The following activities were part of the reading material. I thought it would be best if I complete them as part of my practice:

Think of a recent programming project you've completed. Identify the most prominent objects in the design.

The most recent programming project I have completed is a script that extracts information from a simulation exercise file. The most prominent objects in this exercise file are flights (aeroplanes), the exercise itself, and frequencies (radio frequencies).

What attribute types did they have? Where these attributes actually behaviours?

The attributes of these objects consist of primitive data types. Each flight has a speed, altitude, route, departure aerodrome, arrival aerodrome etc.

Think of a future project. Conduct basic object-oriented analysis and define major and minor objects. Identify the requirements and interacting systems. Sketch out a class diagram. Go into detail about the attributes and methods of some of the most interesting ones. Look for places where you could use inheritance or composition. Look for places where you should avoid inheritance.

My next project will be a booking system which will allow customers to book simulator sessions.

The first objects that come to mind are:

- The simulators
- Exercises

Their attributes would be as follows:

- Simulators: Room Number, Capacity, Type
- Exercises: Type, Difficulty Level, Duration

Those are the first objects and attributes that come to mind.

Requirements: A user (in our case, a person) would have to be able to access the system to make a new booking. If the 
user is already registered with the booking system, they would then select 
the simulator they would like to use, as well as the type of training they would like to conduct, how many students 
they would like to register to this course and what exercise they would like to run. 
In case the user is not yet registered, they would have to make an account and enter their credentials before being able 
to make a booking. After all details have been filled in, the user would select 
a date and time (what we would refer to as a “slot”). If this slot is already taken, the system would inform the user, 
and the user would then be able to choose a different slot. If the slot is available, 
the user can add the booking and the system would notify the supervisor about the new booking. The supervisor would then 
be able to accept or reject this booking.
High-Level Class Diagram of this system. 

In our basic diagram, each class has one method:

- canbook(): This method calls the isavailable() method of the Simulator class (see below). Since our system assumes that only users who are entitled to make bookings have access to it, 
it doesn’t check whether the person is entitled to do so.
- isavailable(): This method is called by the canbook() method and checks whether the simulator is available 
during the requested time slot by comparing the time slot to other bookings or scheduled maintenance activities. 
If the time slot does not match with any of the aforementioned and the requested capacity matches with the total capacity, 
the method returns TRUE.

Abstractions: For our booking system to work, the program doesn’t need to know the setup of each particular simulator, 
nor does it need to know the details of each exercise.
