Unit 11: Artefacts and Reflection

Working on my assignment

Download the assignment files here.

The majority of my time during this unit was spent finalizing my assignment. When I first started working on it, I decided to separate the main operations from each other and assign them to their own Python file. After sketching out the initial classes and methods, I had three Python files:

  • routeplanner.py
  • maintenancemonitor.py
  • energymonitor.py

"routeplanner.py" is responsible for planning routes, as the name suggests. It will receive user input and the car's current location. Based on where it is and where the user wants to go, it would then navigate the car via the shortest route. While planning this system, I came across the A*- as well as Dijkstra's algorithm. After doing some research about these algorithms, I decided to implement Dijkstra's algorithm. I created a "graph" network for a ficticious version of Qatar's capital, Doha, where I currently reside. This graph network would include some main locations the user can navigate to, as well as "weights" or "costs" between them. The algorithm would then find the shortest route and "navigate" the car between current location and destination.

Choosing to include this algorithm in my code cost a lot of time, as I first had to learn how this algorithm works, and how I can customize it to fit my purposes. In the end though, I am happy I chose to include it, as it provides my software with much needed functionality, and it helps to demonstrate how my own functions work and how data is stored within my program.

The other operations

Next to "routeplanner.py", I had created "energymonitor.py" and "maintenancemonitor.py". "energymonitor.py" is responsible for monitoring the car's battery and energy levels. It is also responsible to approve routes calculated by the route planner. If the car doesn't have enough energy at all, the EnergyMontior can instruct the car to move to the closest charging station.

"maintenancemonitor.py" is responsible for monitoring all car parts and their conditions. Whenever the car takes a trip, the MaintenaneMonitor checks each part's deterioration and alerts the user in case a part has deteriorated too much.

Other files

Along the Python files for the three main operations, I created a Python file which hosts all navigational data, such as the "graph" network for Dijkstra's algorithm, as well as a list of charging stations within the city, which is used by the EnergyMonitor. I also created a file which hosts the user interface (userinterface.py). This file not only includes the code for the UserInterface class and its associated methods, but for a small terminal UI which can be used to execute various aspects of my code.

The last steps

Now began one of the main tasks: Testing. I had already used print() statements throughout my ocde to test individual functions. However, I now needed a way to test every Python file. Because I decided to write an individual file for each of the main operations, I had to import classes, objects, and other data between files. This meant I needed a way to continously test all aspects of my code without cluttering my terminal. That is where "assert" statements came in handy. I wrote an assert statement for each class, as well as each statement, and tested my code until it passed all tests without any errors raised. Next, I chose to use "pylint" to test my code design and formatting. Finally, I wrote the "ReadMe" file to dicuss various aspects of my code, and explain how to use it.

Reflection

Overall I am happy with how I tackled this assignment. I definitely think I might have planned much more than I was asked to do, but whenever I am working on a project, I want it to be perfect. There is nothing wrong with that, but this sometimes results in more work than necessary. For example: Did I really have to use Dijkstra's algorithm? Probably not. Does it add value to my project and bring it closer to a real driverless car system? Definitely.

Another aspect I want to improve on is testing. I waited until the end to write my assert statements. This did not help me in any way, it just made it more difficult to test the code later. A few errors and mistakes went unnoticed, and I had to rewrite large chunks of my code. The main cause of this was that I used several python files for my code, not just one. In the future, I plan to write assert statements or other tests while still working on my code. Additionally, I was so focused on the classes and methods of my project, that I completely forgot to write an actual user interface that could be used to run my code, so I had to do that at the end as well. For future projects, I will include the testing in my planning phase, where I would like to sketch out tests for my code, as well as ways to run the code, so I can avoid these issues towards the end.
